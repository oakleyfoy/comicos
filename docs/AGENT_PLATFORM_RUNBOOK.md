# Agent Platform Runbook

This runbook covers the operational behavior of the ComicOS agent platform delivered in P45-01 through P45-07. The platform is deterministic, append-only for execution lineage, permission-gated, and intentionally non-autonomous.

## Scope

- Agent registry and execution lifecycle
- Workflow orchestration lifecycle
- Research and intelligence recommendation flows
- Permission and audit behavior
- Dashboard and analytics usage
- Troubleshooting and recovery

## Operating Principles

- Agent definitions and workflow definitions are deterministic registry assets.
- Execution history is append-only. Completed and failed runs are not rewritten.
- Recommendations are advisory only. No automatic inventory, pricing, or marketplace mutations occur in P45.
- Permission enforcement is deny-by-default.
- Analytics snapshots are generated manually and stored append-only.

## Agent Lifecycle

1. Agent definitions are seeded or registered in the agent registry.
2. Agents remain disabled until explicitly enabled.
3. Execution requires:
   - agent enabled
   - explicit execute permission
   - explicit read permissions for declared capabilities
4. Each run creates an `AgentExecution` row and append-only execution events.
5. Terminal states are `COMPLETED` or `FAILED`.

## Workflow Lifecycle

1. Workflow definitions are seeded or created with deterministic ordered steps.
2. Workflows remain disabled until explicitly enabled.
3. Workflow execution starts in `PENDING` and advances one step at a time.
4. Each step requires:
   - workflow enabled
   - step agent enabled
   - execute permission for the step agent
   - read permission for required capabilities
5. Step denial or execution failure stops forward progress and records failure lineage.

## Permission Model

Supported scopes:

- `read`
- `write`
- `execute`
- `review`
- `admin`

Operational guidance:

- `read` is required for declared capabilities.
- `execute` is required to start an agent run.
- `review` is required to mark recommendations reviewed or dismissed.
- `admin` is required to accept recommendations.
- `write` is intentionally not used for autonomous business mutation in P45.

Audit behavior:

- Denied permission decisions are stored in `agent_permission_audit_event`.
- Policy administration is ops-admin-only.
- Unknown capabilities are denied by default.

## Recommendation Model

- Research agents produce snapshots, findings, and evidence.
- Intelligence agents produce immutable recommendations plus append-only evidence.
- Review activity is stored in separate recommendation review rows.
- Recommendation acceptance is an operator-side acknowledgment only. It does not execute downstream business changes.

## Dashboard Usage

Use `Agent Dashboard` for:

- current agent and workflow health
- recent execution visibility
- recommendation review queue
- analytics snapshot visibility
- platform readiness summary

Expected operator flow:

1. Review platform status.
2. Review agent and workflow health.
3. Inspect recent executions for failures or unusual duration.
4. Review open recommendations and evidence.
5. Generate a fresh analytics snapshot when a point-in-time performance view is needed.

## Analytics Usage

- Analytics are generated manually via `POST /api/v1/agent-analytics/generate`.
- Snapshots are append-only and owner-scoped.
- Metrics cover:
  - agent success and failure rates
  - workflow success and failure rates
  - average execution duration
  - recommendation acceptance and dismissal rates
  - recommendations generated by type

Operational note:

- Lack of a snapshot is a visibility gap, not a mutation risk. Generate a new snapshot before reviewing platform performance.

## Troubleshooting

### Agent run denied

Check:

- agent is enabled
- `agent.execute` policy exists and is allowed
- declared `.read` capability policies exist and are allowed
- recent denied audit events for the agent

### Workflow step fails immediately

Check:

- workflow enabled state
- step agent enabled state
- execute permission for the step agent
- missing read capability permissions
- linked agent execution failure details

### Recommendation review denied

Check:

- `recommendation.review` `review` permission for review/dismiss
- `recommendation.review` `admin` permission for accept
- denied audit events for the underlying agent

### Dashboard shows warning or failure

Check:

- recent failed executions
- latest workflow failures
- missing analytics snapshot
- permission denial spikes

### Analytics status is warning

Common causes:

- no analytics snapshot has been generated yet
- metric counts and stored snapshot summary are out of sync
- recommendation outcome rows are missing for visible recommendations

## Recovery Procedures

### Re-enable a disabled agent

1. Confirm the agent definition is correct.
2. Review recent execution failures.
3. Confirm required read and execute permissions exist.
4. Enable the agent.
5. Perform a controlled manual run.

### Recover from repeated workflow failures

1. Inspect the failed workflow execution and step ordering.
2. Confirm the step agent is enabled and permissioned.
3. Resolve capability-policy gaps.
4. Re-run the workflow manually after validation.

### Recover dashboard or analytics visibility

1. Verify the latest agent dashboard routes respond successfully.
2. Generate a new analytics snapshot.
3. Refresh the dashboard.
4. If visibility still fails, inspect backend test coverage and closeout validation endpoints.

## Readiness Endpoints

- `GET /api/v1/agent-platform/summary`
- `GET /api/v1/agent-platform/validation`
- `GET /api/v1/agent-platform/readiness`

Use these endpoints before production rollout checks, after permission changes, and after significant platform maintenance.
